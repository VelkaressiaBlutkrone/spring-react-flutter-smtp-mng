# 13. 개발 문서 작성 가이드 (Document Writing Guide)

> 프로젝트 전 생애주기에 걸친 문서 작성 방법 정의서

---

## 📁 개발 전 (Pre-Development)

---

### 1. 제안서 (Proposal)

**목적**: 프로젝트의 필요성과 방향성을 이해관계자에게 설득하는 문서

**구성 항목**

- 프로젝트 개요 및 배경
- 문제 정의 및 현황 분석
- 제안 솔루션 및 기대 효과
- 수행 일정 및 예산 계획
- 수행 팀 구성 및 역할

**작성 요령**

- 비전문가도 이해할 수 있는 언어로 작성
- 수치와 근거 데이터를 통해 신뢰성 확보
- 도입 전/후 비교를 시각 자료(표, 그래프)로 제시
- A4 기준 10~20페이지 내외로 간결하게 구성

---

### 2. 계획서 (Project Plan)

**목적**: 프로젝트 수행 범위, 일정, 자원을 체계적으로 관리하기 위한 문서

**구성 항목**

- 프로젝트 목표 및 범위 (Scope)
- WBS (Work Breakdown Structure)
- 마일스톤 및 일정표 (간트 차트)
- 리스크 관리 계획
- 커뮤니케이션 계획 (회의, 보고 주기)

**작성 요령**

- 태스크는 실행 가능한 단위로 분해
- 일정은 버퍼를 포함한 현실적인 기간으로 설정
- 리스크는 발생 확률 × 영향도로 우선순위 정의
- 착수 전 모든 이해관계자의 승인 서명 획득

---

### 3. 요구사항 정의서 (Requirements Definition)

**목적**: 시스템이 수행해야 할 기능과 비기능 요구사항을 명확히 정의하는 문서

**구성 항목**

- 기능 요구사항 (Functional Requirements)
- 비기능 요구사항 (Non-Functional: 성능, 보안, 가용성 등)
- 제약 조건 및 가정 사항
- 요구사항 추적 매트릭스 (RTM)

**작성 요령**

- 각 요구사항에 고유 ID 부여 (예: FR-001, NFR-001)
- 모호한 표현 지양 → "빠르게" 대신 "3초 이내 응답"으로 명시
- 우선순위 분류: Must Have / Should Have / Nice to Have
- 고객 확인 후 버전 관리(v1.0, v1.1)로 변경 이력 관리

---

### 4. 아키텍처 설계서 (Architecture Design)

**목적**: 시스템의 전체 구조와 기술 스택, 컴포넌트 간 관계를 정의하는 문서

**구성 항목**

- 아키텍처 개요 및 설계 원칙
- 시스템 구성도 (전체 구성도, 배포 구성도)
- 기술 스택 정의 (언어, 프레임워크, 인프라)
- 컴포넌트 간 인터페이스 정의
- 보안, 확장성, 가용성 설계 방안

**작성 요령**

- C4 모델(Context → Container → Component → Code) 레벨별 다이어그램 활용
- 설계 결정 사항마다 선택 이유(Trade-off) 기술
- 다이어그램은 draw.io, Lucidchart 등 도구로 표준화
- 개발팀과 함께 검토 후 최종 확정

---

### 5. ERD (Entity Relationship Diagram)

**목적**: 데이터베이스의 테이블 구조와 관계를 시각적으로 정의하는 문서

**구성 항목**

- 엔티티 목록 및 속성 정의
- 관계 (1:1, 1:N, N:M) 및 참조 무결성
- 인덱스 설계 방안
- 테이블 정의서 (컬럼명, 타입, 제약조건, 설명)

**작성 요령**

- 논리 ERD → 물리 ERD 순서로 작성
- 명명 규칙 통일 (snake_case 권장, 테이블 접두사 등)
- N:M 관계는 중간 테이블로 분해하여 표현
- dbdiagram.io, ERDCloud 등 협업 도구 활용 권장

---

### 6. API 명세 (API Specification)

**목적**: 서버와 클라이언트 간 인터페이스를 명확히 정의하여 병렬 개발을 가능하게 하는 문서

**구성 항목**

- API 기본 정보 (Base URL, 인증 방식, 버전)
- 엔드포인트 목록 (Method, Path, 설명)
- 요청 파라미터 (Header, Query, Path, Body)
- 응답 형식 (Status Code, Response Body, 에러 코드)
- 요청/응답 예시 (JSON 샘플)

**작성 요령**

- OpenAPI(Swagger) 표준 스펙으로 작성하여 자동화 활용
- 에러 응답도 성공 응답과 동일한 수준으로 상세 기술
- 필드별 타입, 필수 여부, 유효성 조건 명시
- Postman Collection으로 테스트 가능하게 제공

---

## 🛠️ 개발 중 (During Development)

---

### 7. 상세 설계서 (Detailed Design)

**목적**: 아키텍처 설계를 기반으로 각 모듈의 내부 로직을 구체적으로 설계하는 문서

**구성 항목**

- 클래스/모듈 설계 (클래스 다이어그램)
- 주요 기능별 시퀀스 다이어그램
- 알고리즘 및 핵심 로직 기술
- 공통 컴포넌트 및 유틸리티 정의

**작성 요령**

- UML 표준 다이어그램 적극 활용
- 복잡한 비즈니스 로직은 플로우차트로 보완
- 개발자가 보고 바로 코딩 가능한 수준으로 상세화
- 설계 변경 시 즉시 문서 반영하여 코드와 동기화 유지

---

### 8. 작업 로그 (Work Log)

**목적**: 개발 진행 상황과 이슈를 일별로 추적하여 프로젝트 가시성을 확보하는 문서

**구성 항목**

- 날짜 및 작업자
- 완료 작업 목록
- 진행 중 작업 및 예상 완료일
- 이슈/블로커 및 해결 방법
- 다음 작업 예정 사항

**작성 요령**

- 매일 EOD(업무 종료 시) 작성하는 습관 형성
- JIRA, GitHub Issues 등 이슈 트래커와 연동 권장
- 블로커는 즉시 팀에 공유하여 지연 최소화
- 주간 단위로 요약본을 별도 작성하여 관리자 보고에 활용

> **참고**: AI 작업 결과물의 Working Log는 [12-working-log.md](12-working-log.md) 규칙을 따른다.

---

### 9. 변경 요청서 (Change Request)

**목적**: 개발 중 발생하는 요구사항 변경을 공식적으로 관리하여 범위 크리프를 방지하는 문서

**구성 항목**

- 변경 요청 ID 및 요청일
- 변경 요청자 및 영향 범위
- 변경 전/후 내용 비교
- 일정 및 비용 영향 분석
- 승인자 서명 및 결정 사항

**작성 요령**

- 구두 변경 요청도 반드시 문서화 후 승인 프로세스 진행
- 변경 영향도(일정, 비용, 품질)를 구체적 수치로 제시
- 승인 전까지 개발 작업 미착수 원칙 준수
- 변경 이력은 버전별로 누적 관리

---

### 10. 테스트 케이스 (Test Case)

**목적**: 기능이 요구사항대로 동작하는지 검증하기 위한 테스트 시나리오를 사전 정의하는 문서

**구성 항목**

- 테스트 케이스 ID 및 분류 (단위/통합/E2E)
- 테스트 대상 기능 및 요구사항 ID 매핑
- 사전 조건 (Precondition)
- 입력값 및 실행 절차 (Steps)
- 예상 결과 (Expected Result)

**작성 요령**

- 요구사항 정의서의 항목과 1:1로 추적 가능하게 작성
- 정상 케이스(Happy Path)와 예외 케이스(Edge Case) 모두 포함
- 경계값(Boundary Value) 및 동치 분할 기법 적용
- TestRail, Zephyr 등 테스트 관리 도구 활용 권장

---

## 📦 개발 후 (Post-Development)

---

### 11. 테스트 보고서 (Test Report)

**목적**: 테스트 수행 결과를 종합하여 시스템의 품질 수준과 출시 가능 여부를 판단하는 문서

**구성 항목**

- 테스트 범위 및 환경
- 테스트 수행 결과 요약 (총 케이스 수, 성공/실패/블로킹)
- 결함(Defect) 목록 및 심각도 분류
- 미해결 이슈 및 처리 계획
- 품질 수준 평가 및 출시 권고 의견

**작성 요령**

- 수치 중심의 객관적 결과 기술 (통과율, 결함 밀도 등)
- 심각도 분류: Critical / Major / Minor / Trivial
- 결함은 재현 절차와 스크린샷을 포함하여 상세 기록
- 이해관계자 리뷰 후 Go/No-Go 결정의 공식 근거로 활용

---

### 12. 사용자 매뉴얼 (User Manual)

**목적**: 최종 사용자가 시스템을 올바르게 사용할 수 있도록 안내하는 문서

**구성 항목**

- 시스템 개요 및 주요 기능 소개
- 시작하기 (로그인, 초기 설정)
- 기능별 사용 방법 (단계별 설명 + 스크린샷)
- 자주 묻는 질문 (FAQ)
- 오류 메시지 및 해결 방법

**작성 요령**

- 기술 비전문가를 대상으로 쉬운 용어 사용
- 각 단계별 스크린샷 또는 GIF 이미지 필수 포함
- 번호가 매겨진 순서형 절차로 작성
- 개발자가 아닌 실제 사용자가 검토 후 피드백 반영

---

### 13. 배포 가이드 (Deployment Guide)

**목적**: 시스템을 운영 환경에 안전하게 배포하기 위한 절차를 정의하는 문서

**구성 항목**

- 배포 환경 요구사항 (OS, 미들웨어, 리소스)
- 배포 전 체크리스트 (백업, 공지, 점검 항목)
- 단계별 배포 절차 (명령어 포함)
- 환경 변수 및 설정 파일 관리 방법
- 롤백 절차 및 배포 검증 방법

**작성 요령**

- 명령어는 코드 블록으로 복사·붙여넣기 가능하게 작성
- 신규 인원도 수행 가능한 수준의 상세한 단계별 기술
- 각 단계별 예상 결과와 오류 발생 시 대응 방법 병기
- 배포 수행 후 체크리스트로 검증하는 절차 포함

---

### 14. 운영 매뉴얼 (Operations Manual)

**목적**: 운영팀이 시스템을 안정적으로 유지하고 장애 발생 시 신속히 대응할 수 있도록 하는 문서

**구성 항목**

- 시스템 구성 및 인프라 현황
- 정기 모니터링 항목 및 임계치 기준
- 정기 작업 목록 (백업, 로그 정리, 패치 등)
- 장애 유형별 대응 절차 (Runbook)
- 비상 연락망 및 에스컬레이션 체계

**작성 요령**

- 장애 대응 절차는 의사결정 플로우차트로 시각화
- 각 절차에 예상 소요 시간 명시
- 운영자가 직접 참여하여 작성하거나 검토
- 분기 1회 이상 정기 업데이트 및 실제 장애 후 회고 반영

## 📌 문서 관리 공통 원칙

| 원칙 | 내용 |
|------|------|
| **버전 관리** | 모든 문서에 버전 번호(v1.0, v1.1)와 변경 이력 기재 |
| **승인 체계** | 주요 문서는 작성자 → 검토자 → 승인자 단계 준수 |
| **저장 위치** | Confluence, Notion, SharePoint 등 중앙 저장소 단일화 |
| **명명 규칙** | `[프로젝트명]_[문서유형]_v[버전]_[날짜].확장자` 형식 사용 |
| **접근 권한** | 문서 민감도에 따라 열람/수정 권한 차등 부여 |
